# 沉浸式专注陪伴工具：可直接投喂 Codex 的完整 Prompt

```text
你是「资深产品经理 + 技术架构师 + 数据分析负责人 + 交互设计师」。
请你为我设计并输出一个可落地开发的「沉浸式专注陪伴工具」完整方案。

# 一、你的任务目标
请围绕下述产品概念，输出“可以直接进入开发”的文档级结果，而不是泛泛建议：
- 核心概念：番茄钟专注系统 + 白噪音/音乐环境 + 虚拟人物陪伴养成 + 笔记系统 + 习惯追踪器。
- 产品原则：
  1) 专注优先，互动为辅助；
  2) 减少孤独感，营造“有人陪我一起努力”；
  3) 轻养成，不重游戏化，不制造焦虑；
  4) 所有功能必须可验证、可埋点、可迭代。

# 二、目标用户与使用边界
- 目标用户：18-35 岁学生、考证人群、知识工作者、自由职业者。
- 典型痛点：拖延、分心、独自学习孤独、难以长期坚持。
- 使用平台（MVP）：Web + PWA（桌面浏览器与手机浏览器均可使用）。
- 不在首版范围（Out of Scope）：
  - 多人实时房间
  - 语音通话/视频
  - 复杂社交系统
  - 超重剧情 RPG 化系统

# 三、核心功能需求（必须覆盖）
1) 专注系统（番茄钟）
- 支持自定义专注时长：25/45/60 分钟（以及用户自定义）。
- 支持休息时长：5/10 分钟（以及用户自定义）。
- 模式：标准模式、深度模式。
- 每次完成专注后获得积分与好感度。
- 支持中断判定、完成判定、失败判定。

2) 音乐与环境系统
- 白噪音：雨声、咖啡馆、森林等。
- 音乐：轻音乐、Lo-fi。
- 支持独立音量控制与混音比例。
- 支持保存个人预设（如“雨夜学习”“清晨写作”）。

3) 虚拟人物 A 陪伴系统（核心亮点）
- 人物 A 在主界面持续陪伴（可静默或隐藏）。
- 专注开始时触发鼓励文案。
- 休息阶段触发简短互动对话（不打断专注过程）。
- 根据好感度等级（Lv1~Lv10）解锁不同语气、内容、剧情片段。
- 好感度主要由有效专注时长累计。
- 不同等级体现不同性格表现与互动方式。

4) 笔记系统
- 专注期间可快速记录想法。
- 自动关联当前番茄记录。
- 支持标签、检索、按时间筛选。

5) 习惯追踪系统
- 用户可设定长期目标（如每周专注总时长、每日番茄数）。
- 可视化统计（日/周/月）。
- 连续打卡奖励机制。

# 四、数值与规则（首版固定参数）
- 每完成 25 分钟有效专注：+10 积分、+10 好感。
- 连续完成奖励：
  - 连续 1~3 个番茄：倍率 1.0
  - 连续 4~6 个番茄：倍率 1.2
  - 连续 7 个及以上：倍率 1.3
- 深度模式奖励系数：1.1
- 中断规则：
  - 中断超过 3 分钟，判定本次番茄失败；
  - 失败仅发放 20% 积分，不发放剧情奖励。
- 防刷机制：
  - 每日好感上限 200；
  - 同日高频重复刷同一短时长时，从第 6 次起奖励 *0.8。
- 特殊触发：连续 7 天每天至少完成 1 个番茄，触发特殊剧情。

# 五、体验与伦理要求（必须严格执行）
- 文案风格：温和鼓励、支持型，不使用羞辱/PUA/负罪感表达。
- 用户可关闭互动，仅保留纯专注模式。
- 虚拟角色需透明提示“为 AI/系统生成内容”。
- 连续失败时不给惩罚文案，只提供恢复建议。
- 晚间（可配置）减少强刺激互动，避免影响睡眠。

# 六、技术与实现约束
- 前端：Next.js + TypeScript + TailwindCSS。
- 后端：Supabase（Auth + Postgres + Storage + Edge Functions）。
- 数据设计需支持后续 App 化（iOS/Android）迁移。
- 所有关键流程需有埋点事件。
- 方案必须区分：MVP（4 周可交付）与二期扩展。

# 七、你必须输出的内容结构（严格按顺序）
A. 产品 PRD（问题定义、目标、用户故事、非功能需求）
B. 信息架构（IA）与页面流转（文本流程图）
C. MVP 功能清单（P0/P1/P2 分级，含不做清单）
D. 详细交互流程（开始专注、专注中、休息互动、复盘）
E. 虚拟人物养成系统设计（等级、人格维度、状态机、剧情解锁规则）
F. 数据库模型（表结构、字段、索引、关系）
G. API 设计（接口清单、入参/出参、错误码）
H. 前端页面与组件树（页面级+核心组件级）
I. 埋点方案（至少 10 个核心事件，含事件属性）
J. 指标体系（北极星指标 + 漏斗 + 留存 + 质量指标）
K. 验收标准（每个核心模块至少 3 条 Given/When/Then）
L. 测试策略（单元测试、集成测试、E2E、异常场景）
M. 风险与合规（隐私、安全、内容风险、防沉迷）
N. 4 周开发计划（按周里程碑、角色分工、交付件）
O. 二期路线图（功能扩展与技术演进）

# 八、输出质量要求
- 使用中文输出。
- 结构清晰，尽量表格化。
- 明确“可执行动作”，避免空泛描述。
- 所有关键规则必须有数值或阈值。
- 每个模块都要写：目标、输入、输出、边界、异常处理。
- 对关键决策补充“为什么这样设计”。

# 九、验收口径（你必须自检）
在输出最后增加“自检清单”，逐条确认：
1) 是否覆盖五大核心模块；
2) 是否给出 MVP/二期边界；
3) 是否有数据库与 API 草案；
4) 是否有埋点与指标；
5) 是否有 Given/When/Then 验收标准；
6) 是否满足轻养成、低打扰、专注优先原则。

请开始输出最终方案。
```

## 使用说明（给提问者）
- 直接把上面的代码块完整复制给 Codex 即可。
- 若你有特定美术风格（如二次元、极简、像素风），可在“二、目标用户与使用边界”后追加“视觉风格约束”。
- 若你要更快落地，可在 prompt 最前面再加一句：“优先输出 MVP 的数据库与 API，再补产品文档”。
